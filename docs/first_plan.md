本ドキュメントは、Windows環境においてカスタムマウスカーソルを自作・一括管理・共有するための技術要件、デザイン仕様、およびアプリケーションの開発計画をまとめたものである。

---

## 1. マウスカーソルの基本仕様

**ライセンスとビジネスモデル:** 
- アプリ本体は **OSS / MIT ライセンス**で公開（GitHub）。完全無料。
- 収益化は v1.0 ではスコープ外（GitHub Sponsors の任意受付のみ可）。
- フリーミアム（Pro機能の有償化）への将来拡張余地は、コアを MIT のまま別モジュールをクローズドにする形で常に確保しておく。
- EV コードサイニング証明書は、OSS 向け無料署名サービス（SignPath.io 等）の利用を第一選択とする。

**サポート OS（v1.0）:** Windows 10 22H2 以降、および Windows 11 全バージョン。
- Win10 22H2 未満（1809〜21H2）は互換性検証対象外とし、サポートしない。
- Win10 サポートは 2025-10-14 に終了済みのため、より古いバージョンを支える意味は薄いと判断。
- v2.0 以降で Mica / SystemBackdrop / 角丸タイトルバー等の Win11 専用 UI 効果を採用する余地を残す。

Windows OSで最適なユーザー体験を提供するため、以下の形式でアセットを管理する。

- **対応形式:** `.cur`（静止画）、`.ani`（アニメーション）。
  - ※初期リリース(v1.0)では、`.ani` は既存ファイルのインポートのみ対応とし、新規生成は見送る。
- **高DPI・マルチモニター対応:** 
  - 単一の `.cur` ファイル内に複数解像度（**32x32, 48x48, 64x64, 96x96, 128x128, 256x256px** の6サイズ）をパッキングして自動生成する。
  - 96px / 256px を含めるのは、Windows 11 のアクセシビリティ「カーソルサイズ」スライダー（最大15相当 = 約5倍拡大）および 4K/8K + 200%スケーリング環境で 128px でもボヤけるため。
  - これにより、Windowsの拡大率（100%、150%など）に応じてOSが最適な解像度を自動選択し、常にピクセルパーフェクトな画質を維持する。
- **重要設定:** **ホットスポット（Hotspot）**。画像のどの座標をクリック判定とするかを必ず定義する。
  - 縮小時の丸め誤差を防ぐため、メタデータ（`theme.json`）上では常に「元画像の絶対ピクセル値（整数）」で保持する。

## 3. アプリケーション開発要件

Windowsのネイティブ機能を最大限活用するため、以下の技術スタックとアーキテクチャを採用する。

### 技術スタックと処理分担（ハイブリッドアーキテクチャ）

- **Framework:** **Tauri v2** (Windowsターゲット)
- **Frontend (Nuxt 4 / Vue.js):** 
  - UI描画、SVG/PNGのインポート読み込み、Canvasでの画像ラスタライズおよびRGBAピクセルバッファの抽出を担当。
- **受け入れる入力画像フォーマット（v1.0）:**
  - **新規制作系:** `PNG`, `SVG`
  - **既存資産取り込み系:** `ICO`, `CUR`, `ANI`
  - 上記5種以外（WebP / JPEG / GIF / TIFF / BMP 等）は v1.0 ではサポートしない。アタックサーフェスと依存ツリーを最小化するため。
  - `ICO` / `CUR` は複数解像度内蔵構造を解析し、内部画像を6サイズスロットに自動マッピングする。
  - `ANI` は §「アニメーションカーソル（`.ani`）のプレビュー制御」のロジックでパース。新規生成は v1.0 ではサポートしない。
- **Backend (Rust / `windows-rs`, `winreg`):**
  - **Source of Truth:** アプリケーションの設定（`config.json`）はRustが保持・一元管理し、UIが閉じられていてもバックグラウンドで動作可能にする。
  - **画像処理:** Nuxtから受け取ったRGBA配列を元に、高品質な縮小処理（Lanczos法等）と `.cur` バイナリフォーマットのパッキングをRustで行う。
  - **ファイルI/O:** IPC通信のボトルネックを避けるため、`.cursorpack` のZip解凍やファイルのディスクへの書き込み処理は全てRust側に丸投げする。

### 実行モデル・配布形態・ファイル保存戦略

- **常駐型アプリ:** システムトレイ（タスクトレイ）に常駐し、バックグラウンド処理を行う。
- **OS起動時のサイレント自動起動:** `HKCU\Software\Microsoft\Windows\CurrentVersion\Run` レジストリにアプリを登録し、PC起動時にメイン画面を出さずにバックグラウンド（トレイのみ）で自動起動させることで、常時自動ダークモード切替を機能させる。
- **超低メモリ常駐設計 (エコ仕様):** ユーザーがメイン画面を閉じてトレイに格納した際は、メモリを大量に消費する WebView エンジン（Nuxtの描画層）を完全に破棄（Drop）する。ダークモード監視などは数MBで動くRustのネイティブループ単独に任せることで、常駐時のRAM使用量を極限まで削減する。
- **イベント駆動の監視（ダークモード判定）:** 
  - 起動時に `HKCU\Software\Microsoft\Windows\CurrentVersion\Themes\Personalize` の `AppsUseLightTheme`（DWORD: 0=ダーク / 1=ライト）を読み込んで初期状態を確定。
  - 以降は Rust のネイティブループで `WM_SETTINGCHANGE` メッセージを購読し、`lParam` が文字列 `"ImmersiveColorSet"` の通知を受けたタイミングで上記レジストリを再読込する。
  - WinRT (`UISettings.ColorValuesChanged`) は採用しない。WinRT ランタイム初期化を避けることで「数MB常駐」目標と整合させる。
  - 依存クレートは `winreg`（レジストリ）+ `windows-rs`（Win32 メッセージループのみ）に限定。
- **UAC（管理者権限）不要のクリーン設計:** ファイルの保存先をユーザー固有の `~/.custom_cursors/` に、レジストリ操作も `HKCU` に限定することで、日常利用において「管理者権限の許可（UACプロンプト）」を一切要求しない安全で煩わしさのない設計を保証する。
- **配布とストレージ設計（MSIX互換フェイルセーフ設計）:** 
  - 初期は `.msi`、将来は `.msix` での配布を想定。
  - **カーソル保存先:** アンインストール時の強制削除を回避するため、カーソルの実体は `AppData` ではなく、ユーザーディレクトリ直下（`C:\Users\[UserName]\.custom_cursors\<UUID>\`）に隔離保存する。
  - **アンインストール挙動:** アプリをアンインストールした場合、アプリ本体と `AppData/Local/[AppID]`（設定データ）はクリーンアップされるが、`~/.custom_cursors/` とWindows標準のスキーム一覧（レジストリ）はあえて残し、引き続きWindows標準機能としてカーソルを使えるように保護する。
  - **MSIX レジストリ仮想化対策（重要）:** MSIXは既定で `HKCU` への書き込みを Package Family 配下に仮想化するため、`HKCU\Control Panel\Cursors` への書き込みが OS本体に反映されない事態が発生し得る。これを回避するため、MSIX 配布時は `runFullTrust` capability を宣言した **Packaged Win32 App** として構成し、レジストリ書き込みをパススルーさせる。Microsoft Store 提出時は審査が厳しくなる点を承知の上で計画する。
  - **MSIX スタートアップ:** MSIX 環境では `HKCU\...\Run` への直接登録は非推奨のため、AppxManifest の `<Extension Category="windows.startupTask">` で自動起動を宣言する。`.msi` 版は従来通り `HKCU\...\Run` を使用する。
  - **WebView2 ランタイム依存:** Tauri v2 は WebView2 が必須。Windows 11 は標準同梱だが、Windows 10 1809 未満や LTSC では別途インストールが必要。`.msi` には Evergreen Bootstrapper を同梱し、初回起動時にサイレント取得する戦略をとる。
  - **コードサイニング:** 常駐型かつ自動起動する性質上、未署名では SmartScreen で初回ブロックされ致命的。OV/EV 証明書を取得し、最低限 OV、可能なら EV 証明書で署名する（EV はレピュテーション獲得が即時のため推奨）。

### テーマライブラリ管理

- **整理機能（v1.0）:** ソート + フリーテキスト検索 + お気に入り（スター）の3点に絞る。
  - **ソート軸:** 名前順 / 更新日順 / 適用回数順
  - **検索:** テーマ名・作者名のフリーテキスト部分一致
  - **お気に入り:** スター付与で一覧トップに固定
- **タグ / フォルダによる階層分類は v1.0 ではスコープ外**。テーマが激増した場合の対応として将来追加余地を残すが、初期はフラット一覧を維持。

### テーマプレビュー

- **静的カードプレビューのみ:**
  - テーマ一覧 UI では各テーマカード上に**17役割の縮小画像（48px相当）**をグリッド表示し、見た目の確認を可能にする。
  - **ライブプレビュー（OS全体への一時適用 + 自動復帰）は v1.0 ではスコープ外**。WebView内インラインプレビューも実装しない（ホットスポット未対応・DPI問題・`.ani`非対応のため労力対効果が悪い）。
  - 実環境での確認は「適用してダメならパニックボタンで Windows 既定に戻す」フローに統一する。

### 主要機能（UI / UX）

- **二面性のあるUI設計:**
  - **一般ユーザー向け（メイン画面）:** 極めてシンプル。`.cursorpack` をドラッグ＆ドロップして「適用」するだけ。
  - **クリエイターモード（編集画面）:** **解像度別アサイン方式**を採用。
    - 1役割につき、6サイズ（32/48/64/96/128/256px）それぞれに**独立した画像**を割り当て可能（割り当てなしのサイズは元画像から自動リサイズ）。
    - これによりドット絵作家が「32pxはピクセルアート、128pxは別途滑らかな版」を使い分けられる。
    - **ホットスポットも解像度別に指定可能**（必須ではない／未指定時は基準サイズから比例計算）。32px と 256px の1pxズレは視覚的影響が大きいため、解像度別の微調整を許容する。
    - 簡易ペイント／ベクター編集は v1.0 のスコープ外。作画は外部ツール（Aseprite / Photoshop / Inkscape 等）に委ね、本アプリは**管理と適用**に専念する。
- **モダンな割り当て操作:**
  - 17種の役割スロットに対して画像をドラッグ＆ドロップで割り当てるグリッドUI。
  - **賢いオートマッピング:** 単なる完全一致ではなく、Rust側にキーワード辞書（例: wait, busy, loading）を持たせ、ファイル名から柔軟に役割を推測・自動配置するファジーマッチングを実装。

### データ管理・テーマ共有

- **テーマ共有パッケージ（`.cursorpack`）:**
  - 画像群とメタデータ（`theme.json`）をまとめたZipベースの独自パッケージ形式。
  - メタデータ内には「OS標準の影を必要とするか（`requires_os_shadow`）」のフラグを持たせ、画像に焼き付けられた影との競合（二重の影）を防ぐ。
- **役割の最低必須要件（部分テーマ対応）:**
  - **`Arrow`（通常の選択）のみ必須**。残り 16 役割は任意。
  - **未指定の役割スロットは適用時に Windows 標準を継承**する。レジストリ書き込みでは対応値を空文字列または未設定として書き出し、`Schemes` の文字列フォーマット中の空エントリ仕様に従って OS が既定カーソルにフォールバックする。
  - これは既存コミュニティテーマ（Arrow + Wait + IBeam + Hand 程度のみ収録するものが多数派）への互換性を確保するための判断。
  - **トレードオフ:** 「あえて Help を空にしたい」という意図を技術的に区別できない（未指定 = 標準継承）。この制限はユーザー向けドキュメントに明記する。
  - メイン画面のテーマカード UI では、**未指定スロットを Windows 既定アイコンの薄表示**で示し、テーマがどのスロットを上書きし、どのスロットを継承するかを一目で分かる表現にする。
  - 公式インデックス提出時の CI 検証は「Arrow が含まれているか」のみ必須チェック。他の役割は欠落していても警告にとどめる。
- **`theme.json` 必須/推奨フィールド:**
  - 必須: `schema_version`, `id`(UUID), `name`, `version`(SemVer), `created_at`, `requires_os_shadow`, `cursors`(役割→ファイル/ホットスポット マップ)
  - 推奨: `author`, `license`(SPDX識別子), `homepage`, `description`, `min_app_version`, `signature`(将来の署名検証用)
  - 多言語対応: `name` / `description` は `{ "ja": "...", "en": "..." }` の i18n オブジェクトを許容。
- **同名テーマ再インポート時の挙動:**
  - 既存テーマと同 `id` の場合は `version` を比較し、上書き / バージョン併存 / 別名保存をユーザーに選ばせるダイアログを出す。`id` 衝突は UUID 生成側の問題として明示的にエラー表示。
- **配布エコシステム（公式インデックス + 任意URLインポート）:**
  - **方針:** 独自バックエンドは持たず、**公式メタデータインデックス（GitHub 上の単一 JSON）** を Source of Truth とする。実体ファイルは GitHub Releases / 個人サイト等のサードパーティに置く。
  - **インデックス構造（例）:** `id`(UUID) / `name` / `author` / `homepage` / `download_url` / `sha256` / `signature` / `author_pubkey_id` / `tags`。
  - **署名と真正性:**
    - **公式掲載テーマ:** 作者公開鍵（Ed25519）による署名検証を**必須**。インデックスに登録済みの公開鍵で `download_url` の SHA-256 を署名する方式。検証失敗 → インストール拒否。
    - **任意URL／ローカルファイルからのインポート:** 署名なしを許容するが、初回インポート時に「未署名テーマです。信頼できる出所か確認してください」の**強警告ダイアログ**を表示し、ユーザーの明示的同意を取得する。
  - **モデレーション:** インデックスは GitHub の Pull Request ベースで運用。レビュー/CI で形式検証＋既知マルウェアハッシュとの突合を行う。
  - **将来の拡張余地:** Web 化（独自ストア）への移行を妨げないよう、インデックススキーマはバージョニング（`schema_version`）し、後方互換を維持する。
- **作者の公開鍵管理（Ed25519 署名運用）:**
  - **鍵生成:** クリエイターモード内の専用UIで鍵ペアを生成。秘密鍵は `~/.custom_cursors/_keys/private.key` に **Windows DPAPI（`CryptProtectData`）でユーザーアカウント縛り暗号化**して保存。公開鍵は同フォルダに `public.key` として平文保存。
  - **秘密鍵のバックアップ / 移行:** クリエイターモードから「秘密鍵をエクスポート（パスフレーズ付き）」「秘密鍵をインポート」を提供。PC 移行時に過去テーマの再署名が可能。
  - **鍵 ID:** 公開鍵の SHA-256 の先頭 16 文字を `key_id` として使用。テーマメタデータには `key_id` のみ記録、フル公開鍵は公式インデックスから引く（テーマ容量削減）。
  - **公開鍵の登録方法:** 公式インデックスリポジトリの `authors/{github_username}.json` に公開鍵を登録する PR を提出する形式。GitHub アカウントを身元証明として活用し、独自認証基盤は持たない。
  - **鍵ローテーション（紛失・漏洩時）:** 新公開鍵で `authors/{github_username}.json` を上書きする PR を提出。**過去テーマの再署名は不要**（インデックスは過去 `key_id` の検証情報も保持し続けるため、古い鍵で署名されたテーマも検証可能）。鍵紛失時は GitHub アカウント所有を運営が確認した上で承認。

- **クリエイターからの公式インデックス提出フロー:**
  1. クリエイターがクリエイターモードでテーマを作成 → 「公開用にエクスポート」ボタンで `.cursorpack` + メタデータ JSON（インデックスエントリの雛形）を生成。
  2. 同時にアプリが**ブラウザで公式リポジトリの "Submit Theme" PR テンプレートページ**を開く（GitHub の事前 URL パラメータでテンプレートを埋める）。
  3. クリエイターは GitHub のブラウザ UI から `.cursorpack` をアップロード、説明文を記入して PR を提出（**Git CLI 不要、GitHub アカウントのみ必要**）。
  4. 公式リポジトリ側に GitHub Actions の CI を用意し、以下を**自動検証**して合致しない PR は自動拒否：
     - メタデータ JSON のスキーマバリデーション
     - `.cursorpack` の SHA-256 と提出値の一致
     - 署名（Ed25519）の検証
     - ファイルサイズ閾値チェック（§セキュリティ参照）
     - 既知マルウェアハッシュとの突合
  5. 人間レビュー（運営）は CI 通過した PR の名前重複・著作権侵害・不適切コンテンツのみを目視確認 → マージで公式掲載完了。
- **OAuth 連携 / アプリ内自動 PR 作成は v1.0 ではスコープ外**。GitHub アカウントの認可ハンドリング、トークン保管、Fork 管理の複雑度を避けるため、ブラウザ経由の手動提出を前提とする。
- **ダークモードのペアリング:**
  - `.cursorpack` 側に特別な仕様を設けるのではなく、設定画面でユーザーが「ライト用にはテーマA」「ダーク用にはテーマB」と任意の2つのテーマを自由に紐付けできる設計にする。
- **自動切替トリガー（スコープ確定）:**
  - v1.0 では**自動切替トリガーはダークモード連動の1種類のみ**とする。
  - 時刻スケジュール / アプリフォーカス連動 / グローバルホットキーによるテーマ切替などは**スコープ外**。設計の単純さを優先し、機能追加は v2.0 以降にユーザー需要を見て判断する。
  - グローバルホットキーは「パニックボタン（Windows 既定への復旧）」のみに限定。

### カーソル適用と復旧ロジック

- **レジストリ統合と一括適用:** 
  1. `Set-ItemProperty` 相当のRust処理で `HKCU\Control Panel\Cursors` を書き換え。
  2. Windows標準のスキーム一覧（`HKCU\Control Panel\Cursors\Schemes`）にもテーマを登録し、コントロールパネルからも選べるようにする。
     - `Schemes` の値は **17パスをカンマ区切りで連結した単一の REG_SZ / REG_EXPAND_SZ** であるため、パスに `,` を含めない／環境変数 `%USERPROFILE%` を使う場合は REG_EXPAND_SZ で書き込む等、フォーマット制約を厳守する。
     - 全体長がレジストリ値の上限に達しないよう、UUID 配下のディレクトリ構造を浅く保つ。
  3. `SystemParametersInfoW(SPI_SETCURSORS, ...)` を実行し、再起動なしで即時反映。
  4. テーマのメタデータに基づき `SystemParametersInfoW(SPI_SETCURSORSHADOW, ...)` を呼び出し、Windows標準の「ポインターの影」設定も自動でON/OFF制御する。
- **適用トランザクション（ロールバック機構）:**
  - 17スロットの書き換え途中でエラーが発生した場合の中途半端な状態を防ぐため、適用前に対象レジストリ値を全てスナップショット → 失敗時に自動復元する **疑似トランザクション機構**をRust側に実装する。
  - このスナップショットは「適用処理失敗時の自動復旧」専用とし、**ユーザー操作による Undo（直前テーマへの戻し）機能は v1.0 ではスコープ外**とする。日常的な復旧手段はパニックボタン（Windows 既定への強制リセット）のみに統一する。
  - **クラッシュ・OS 強制終了対策（ディスクスナップショット）:**
    - 適用処理は必ず以下の3ステップで実装：
      1. スナップショット JSON を `~/.custom_cursors/_pending_apply.snapshot` にディスク永続化
      2. レジストリ書き換え + `SystemParametersInfoW(SPI_SETCURSORS)`
      3. 完了したら `_pending_apply.snapshot` を削除
    - スキーマ: `{ schema_version, original_values: { "Arrow": "C:\\...", ... }, applied_at: ISO8601, target_theme_id }`
    - **次回起動時にこのファイルが残っていれば「前回処理が中断された」と判定**し、`original_values` を用いて自動復元する。電源断 / BSOD / Rust panic / OS 強制終了すべてに対応可能。
    - 復元実施時は Windows トースト通知で「前回の中断を復元しました」と告知（通知 UX マトリクスの層2）。
  - panic_hook 内でのレジストリ書き戻しは行わない（panic 中の書き込みは信頼できないため。ディスクスナップショット + 次回起動時復元で代替）。
- **OS設定との状態同期:**
  - ユーザーがコントロールパネルや「マウスのプロパティ」から外部でカーソルを変更した場合、アプリ内部状態と実レジストリが乖離する。`WM_SETTINGCHANGE`（`SPI_SETCURSORS` 関連）を購読して外部変更を検知し、UIに反映する。
- **孤児カーソル復旧（ヘルスチェック）:**
  - ユーザーが `~/.custom_cursors/` を手動削除した場合、レジストリには無効パスが残り**カーソルが消失/真っ白化**する。アプリ起動時にレジストリ参照先のファイル存在を検証し、欠落していれば標準カーソルへ自動復旧する。
- **確実な復旧機能（パニックボタン）:**
  - ワンクリックで Windows 既定（System Scheme）へ強制リセットし、トラブル時でも確実に復旧できる機能を実装。
  - UI 上の表記は最新の Windows 11 公式名称「**Windows 既定**」に統一する（旧称「Aero標準」は俗称のため使用しない）。
  - グローバルホットキー（例: `Ctrl+Alt+Shift+R`）にもバインド可能とし、カーソル消失時の最終救済として常に発動可能にする。
  - 復旧の選択肢は2段階用意する：
    1. **「Windows 既定」へリセット**（OS 出荷時の標準カーソル）
    2. **「インストール前の状態」へリセット**（後述の初期スナップショットから復元）
- **初回インストール時の既存設定スナップショット:**
  - 初回起動時に `HKCU\Control Panel\Cursors` の現状を `~/.custom_cursors/_initial_snapshot.cursorpack` として自動保存する。
  - これにより、本アプリインストール前に他ツール（CursorFX / 自作スクリプト等）で設定していたカーソル構成へいつでも戻せる。
  - アンインストール時にもこのスナップショットは保持し、ユーザーが手動で復元可能にする。
  - **他のカーソル管理ツールの検知・警告は v1.0 ではスコープ外**。共存は黙認しユーザー責任とするが、初期スナップショットによって「最悪の場合元に戻せる」保険を提供する。

### セキュリティとエッジケース対策

- **堅牢なセキュリティバリデーション (ファイル偽装・脆弱性対策):**
  - **Magic Byte (ファイルシグネチャ) 検証:** コミュニティで配布されたテーマ内に悪意のあるファイル（例: `.exe` を `.cur` に偽装）が混入しているケースを防ぐため、拡張子だけでなくファイルの先頭バイト（Magic Bytes）を読み込み、真のファイルフォーマットを厳格に判定・拒否する。
  - **パストラバーサル・Zip爆弾対策:** 解凍パスに `../` などのディレクトリトラバーサル攻撃が含まれる場合や、展開後のファイルサイズが異常に大きい（Zip爆弾）場合は、Rustの型安全なパーサーでエラーとして安全に弾く。
  - **具体的な閾値（v1.0 既定値、`config.json` でユーザー上書き可）:**
    - `.cursorpack` 圧縮時サイズ上限: **50 MB**
    - 解凍後合計サイズ上限: **200 MB**（解凍は逐次サイズチェックを行い、超過した時点でストリーム中断。メモリ全展開後の OOM を防ぐ）
    - 個別画像ファイルサイズ上限: **10 MB**
    - ユーザー全体テーマ保存量（`~/.custom_cursors/` 合計）: **無制限。1 GB 超過時にトレイ通知で警告**し、テーマ削除UIへ誘導する（強制削除はしない）。
  - **レジストリ・インジェクション対策:** `theme.json` から読み込んだテーマ名をレジストリキーとして書き込む際、バックスラッシュ（`\`）やNullバイト（`\0`）による意図しない階層の上書きを防ぐため、英数字や安全な記号のみにサニタイズする。
  - **シンボリックリンク攻撃の無効化:** Zip解凍時に、シンボリックリンクなどの特殊なファイルが含まれているパッケージは、システムファイル上書きの危険があるため全体をエラーとして拒否する。
  - **画像メタデータのパージ:** PNG画像に隠された不正コードやトラッキングデータ（Exif等）を排除するため、Rustでの画像処理時に純粋なピクセルデータのみを抽出し、元のメタデータは全て破棄して `.cur` を生成する。
  - **XSS（クロスサイトスクリプティング）対策:** Nuxt側でテーマ名や作者名を描画する際、Vueの標準機能でエスケープを徹底し、悪意あるスクリプトの実行を防ぐ（`v-html`の禁止など）。
- **ドット絵（ピクセルアート）への配慮:**
  - カーソル画像がドット絵の場合、滑らかな縮小処理（Lanczos等）を行うと輪郭がぼやけてしまう。画像インポート時に「ドット絵（Nearest-Neighbor法）」か「滑らかな画像（Lanczos法）」かを選択・または自動判定させ、最適なリサイズアルゴリズムを適用する。
- **アニメーションカーソル（`.ani`）のプレビュー制御:**
  - Nuxt/WebView環境は `.ani` 形式の描画に対応していないため、インポートされた `.ani` はRust側でパースしてプレビュー用に変換する。
  - **静的プレビュー:** リスト/グリッドUI用に最初の1フレームを PNG として返却。
  - **アニメーションプレビュー（クリエイターモード必須）:** 全フレームと JIFF タイミング情報を Rust から返し、Canvas でフレーム順次描画して再生する。または APNG/GIF 化して返す方式も可。クリエイターは作品確認のために必ず動的再生を要する。
- **SVG インポート時のサニタイズ:**
  - SVG には `<script>` や外部参照（`<image href="http://...">`）が埋め込まれ得るため、Canvas にラスタライズする前に必ず DOMPurify 等で sanitize する。Canvas 経由でも、フェッチタイミング次第で SSRF/情報漏洩の口になり得るため、外部 URL 参照は一律拒否する。

## 4. Windows 11 機能との統合と競合制御

- **アクセシビリティ機能との連携:**
  - Windows 11 の「アクセシビリティ → マウスポインター」では、サイズ／色／反転（`HKCU\Software\Microsoft\Accessibility\CursorIndicator`）と、カスタムカラー（`HKCU\Control Panel\Cursors\ContrastScheme` / `CursorBaseSize` 等）が個別に存在する。
  - 本アプリは**カスタム画像優先**方針とするが、ユーザーが OS 側で色・サイズ変更を行った場合の競合方針を以下のとおり明文化する。
    - サイズ変更: OSスケーリングに従い、6サイズ収録の `.cur` から最適解像度を OS が自動選択（変更を尊重）。
    - 色反転 / カスタムカラー: カスタム画像との同時適用は不可のため、カスタム適用時は警告ダイアログを表示し、ユーザーに「OS設定を一時無効化する／このまま適用する／適用を取り消す」を選択させる。
- **明示的な制限事項（README に記載必須）:**
  - **UAC Secure Desktop**（昇格ダイアログ表示中）では Windows 標準カーソルに戻る。HKCU 限定設計の必然的制約。
  - **ロックスクリーン / サインイン画面**でも Windows 標準カーソルとなる。
  - **マルチユーザー環境**ではユーザーごとに独立した設定（HKCU 限定）。

- **動作環境マトリクス（v1.0）:**
  
  | 環境 | サポート | 補足 |
  |---|---|---|
  | デスクトップ Windows 10 22H2 / Windows 11（x64） | ✅ 動作対象 | 主要ターゲット |
  | Windows on ARM64（Surface Pro X 等） | ✅ 動作対象 | x64 と並行して ARM64 ビルドを v1.0 から提供 |
  | 仮想マシン（VMWare / Hyper-V / VirtualBox ゲスト） | ✅ 動作対象 | VM Tools が独自カーソル描画する場合は OS 設定優先 |
  | タブレット / 2-in-1（タッチ・ペン主体） | ⚠️ 部分対応 | Bluetooth マウス接続時のみ反映。ペン/タッチは別系統で本アプリ非関与 |
  | 非英語ロケール / Insiders / Sandbox | ✅ 動作対象 | 標準 Windows 機能の範囲内 |
  | リモートデスクトップ（RDP / Citrix / RemoteApp） | ❌ 動作対象外 | RDP は独自カーソル描画のため反映不完全。検証コスト過大のためサポートしない |
  | Windows Server | ❌ 動作対象外 | デスクトップ用途想定外。Server Core では Tauri が動作不可 |
  
  - 動作対象外（RDP / Server）は**起動時に検出して警告ダイアログ**を表示し、続行は可能だが期待動作が保証されないことを明示する。

## 5. 配布・運用・パフォーマンス

- **多重起動防止:**
  - 常駐型のため、Named Mutex（`Global\<AppGuid>` または `Local\<AppGuid>`）で多重起動をブロックし、既存インスタンスのトレイアイコンへフォーカス（または通知）を移す。
- **バックアップ / 設定の可搬性:**
  - **フルエクスポート（手動のみ）:** 設定 (`config.json`) と全テーマ実体（`~/.custom_cursors/` 配下）を1つの `.cursorprofile` ファイル（Zip 形式）にまとめてエクスポート可能。
  - **インポート:** `.cursorprofile` を読み込むと、既存設定を上書きするか、テーマだけマージするかを選択可能。`.cursorpack` と同等のセキュリティ検証（パストラバーサル / Zip爆弾 / Magic Byte 等）を適用。
  - **拡張子の使い分け:** `.cursorpack` = 配布用（テーマ単体）、`.cursorprofile` = 個人バックアップ用（設定 + 全テーマ）と明確に区別。
  - 自動定期バックアップ・クラウド同期は v1.0 ではスコープ外。PC 移行 / OS 再インストール / MSIX アンインストール後の復元は、ユーザーが手動で `.cursorprofile` を保管することで担保する。
- **パフォーマンス UX:**
  - 17役割 × 6解像度 = **最大102枚**の Lanczos リサイズが発生するため、適用処理にはプログレス表示とキャンセル機能を実装する。
  - リサイズ結果はキャッシュ化し、未変更スロットの再計算を回避する。

- **パフォーマンス目標値（v1.0、CI で自動測定 + リグレッション検出）:**
  
  | 指標 | 目標値 | 測定タイミング |
  |---|---|---|
  | トレイ常駐時 メモリ使用量（RAM） | **≤ 15 MB** | 起動 10 分後（WebView 破棄済） |
  | メイン画面表示時 メモリ使用量 | **≤ 200 MB** | メイン画面起動 30 秒後 |
  | トレイ常駐時 CPU 使用率 | **≤ 0.1%** | 60 秒平均（平常時） |
  | アプリ起動時間（コールドスタート、トレイのみ） | **≤ 1.5 秒** | OS 起動後の自動起動 |
  | メイン画面起動時間（トレイ → 開く） | **≤ 2 秒** | クリックから描画完了 |
  | テーマ適用時間（17スロット × 6解像度） | **≤ 3 秒** | 「適用」クリックから `SPI_SETCURSORS` 完了 |
  | テーマインポート時間（5MB `.cursorpack`） | **≤ 5 秒** | ドロップから完了 |
  | ダークモード切替の反応時間 | **≤ 500 ms** | `WM_SETTINGCHANGE` 受信から適用完了 |
  | パッケージサイズ（インストーラー、WebView2 ランタイム除く） | **≤ 30 MB** | リリース時 |
  
  - 各値は GitHub Actions の Release ビルドで**自動計測**し、PR ごとにリグレッションを検出する。メモリは手動補完。
  - パフォーマンス CI は **v1.0 から整備**する（後付けでは既存リグレッションが固定化するため）。
  - 目標未達が継続する場合は、根拠（プロファイル結果）と共に仕様書を更新して値を見直す（沈黙的な妥協はしない）。
- **テレメトリ・クラッシュレポート:**
  - クラッシュログは `%LOCALAPPDATA%\<AppName>\logs\` にローカル保存。ネットワーク送信は**ユーザー明示同意（オプトイン）後のみ**実施。送信内容（スタックトレース、OSバージョン、アプリバージョン）を事前明示。

- **ロギング設計:**
  - **クレート構成:** `tracing` + `tracing-subscriber` + `tracing-appender`（Rust エコシステム標準）。構造化ログ（key-value）で記録し、後段の解析容易性を確保。
  - **ログレベル運用:** リリース版は **INFO 既定**。`config.json` で TRACE / DEBUG / INFO / WARN / ERROR を切替可能。トラブル時はユーザーに DEBUG への一時昇格を依頼する運用。
  - **保存先 + ローテーション:**
    - パス: `%LOCALAPPDATA%\<AppName>\logs\app-{YYYY-MM-DD}.log`
    - **日次ローテ + 14 日経過で自動削除**
    - **総容量 100 MB 超過時は古いものから自動削除**
  - **PII（個人情報）除外ポリシー — ログ出力禁止項目:**
    - ファイル絶対パス（ユーザー名を含むため） → 「ユーザーホーム相対パス」に変換して記録
    - レジストリ値の生 RAW（バイナリ含むため） → ハッシュ化 or 構造化された安全な要約のみ
    - `.cursorpack` の SHA-256 そのもの（テーマ匿名性） → 短縮 ID に置換
    - 代わりに記録するもの: 役割名（"Arrow", "IBeam" 等）、アプリバージョン、OS ビルド番号、`event_id`（UUID）。
  - 各ログエントリには `app-version` / `os-build` / `event-id`（UUID）を必ず付与し、ユーザーからのトラブル報告との突合を容易にする。

- **通知 UX（3層分類）:**
  
  | 層 | 用途 | 実装手段 | 該当イベント例 |
  |---|---|---|---|
  | **層1: 無通知** | 期待動作の成功・常態 | — | テーマ適用成功 / ダークモード自動切替 / 定期処理 |
  | **層2: Windows トースト** | 気づきが必要だが操作不要 | `ToastNotificationManager`（Win32 COM 経由で呼び出し、常時 WinRT 初期化はしない） | テーマ適用失敗（ロールバック発動）/ アップデート利用可能 / ストレージ警告（1GB 超過）/ クラッシュリカバリ後の復元通知 |
  | **層3: モーダル** | 操作必要 or 起動阻害 | アプリ内ダイアログ / 起動時エラー画面 | 重大エラー / 未署名テーマインポート時の警告 / マイグレーション失敗 / 動作対象外環境（RDP 等）警告 |
  
  - **AppUserModelID** をマニフェストに登録（MSIX は自動、`.msi` 版はインストーラで設定）し、トースト通知の発信元を明示。
  - **常時 WinRT ランタイム初期化は避ける**（ダークモード監視は Win32 + レジストリで完結する設計と整合）。トースト送出時のみ短時間 COM 初期化する。
  - レガシーなトレイバルーン通知（`Shell_NotifyIcon` の `NIIF_*`）は Win11 で UX が劣化するため使用しない。
- **更新（オートアップデート）:**
  - Tauri Updater を採用。署名検証を必須化し、ダウングレード攻撃を防止。
  - **自動アップデートはユーザー設定で OFF 可能**。常駐型アプリで意に反する更新が走ることを避ける。
  - **アップデート時のユーザーデータ取扱い:**
    - **既存テーマ（`~/.custom_cursors/`）は一切触らない**。アップデートは実行ファイル領域と `AppData/Local/[AppID]/` のみが対象。
    - **設定 (`config.json`) はバージョン番号付きでマイグレーション**。`config.json` 先頭に `schema_version` を保持し、起動時に古ければ最新形式へ変換 → 古い版は `config.bak.v{N}.json` として同フォルダに保管。
    - **マイグレーション失敗時はアプリ起動を中断**して専用エラー画面（リカバリ手順 + `config.bak` の場所案内）を表示。デフォルト設定での強制起動はしない（ユーザー設定喪失を確実に避けるため）。
  - **インストール失敗時のロールバック（トリプルハートビート方式）:**
    - Tauri Updater 標準の差分検証 + 失敗時旧バージョン保持を採用。
    - 新版起動失敗を**3回連続**検出した場合、旧版バイナリへ自動ロールバックし、トレイ通知で告知。
  - **メジャーバージョン跨ぎ（v1 → v2）は自動アップデートしない**。トレイ通知で「v2が出ました。手動でダウンロードしてください」と公式ページへ誘導する二段運用。マイナー (v1.x) は自動。
- **アクセシビリティ（UI側）:**
  - アプリ自身のUIもキーボードナビゲーション、スクリーンリーダー対応、コントラスト比 WCAG AA 以上を満たす。

## 6. 国際化（i18n）

- **アプリ UI:** v1.0 は **日本語 (ja) + 英語 (en) の 2 言語を同梱**。文言は外部リソース化（i18n キー管理）。
  - **デフォルト判定:** OS の `GetUserDefaultLocaleName()` で `ja-*` なら ja、それ以外は en にフォールバック。
  - **手動上書き:** 設定画面で言語を固定可能（OS ロケールに依らず使いたいユーザー向け）。
  - **未翻訳キー検出:** CI でスナップショットテストを実施し、ja / en 間でキー欠落・未翻訳を自動検出。
  - 中国語・韓国語等の追加言語は v1.0 ではスコープ外。リリース後にコミュニティ貢献として受け入れる。
- **テーマメタデータ:** `theme.json` の `name` / `description` フィールドは `{ "ja": "...", "en": "...", "default": "..." }` 形式で多言語化を許可。OS のロケールに応じてフォールバック表示。
- **RTL言語環境:** UI レイアウトの RTL 対応はフェーズ2以降で検討（カーソル画像自体は文化依存のため変換しない）。

## 7. v1.0 MVP リリース基準

### 含まれる機能スコープ

- `.cur` 自動生成（17役割 × 6解像度の Lanczos / Nearest 自動判定）
- `.ani` インポート（新規生成は v1.0 では不可）
- サポート OS: Win10 22H2 + Win11（x64 + ARM64）
- システムトレイ常駐 + WebView 破棄によるメモリ最適化
- HKCU レジストリ統合 + `SystemParametersInfoW` による即時適用
- ダークモード自動切替（テーマ A/B ペアリング）
- クリエイターモード（解像度別アサイン + ホットスポット指定、ペイント機能なし）
- `.cursorpack` インポート / エクスポート + セキュリティ多層防御
- 公式インデックス連携 + Ed25519 署名検証 + 公開鍵管理（DPAPI 暗号化）
- パニックボタン（Windows 既定 / インストール前状態への 2 段階復旧）
- 初回スナップショット + クラッシュリカバリ（ディスク永続化）
- `.cursorprofile` フルバックアップ・エクスポート / インポート
- ライブラリ管理（ソート + 検索 + お気に入り）
- 適用前の静的カードプレビュー
- ja + en 同梱（OS 自動判定 + 手動上書き）
- `tracing` ベースのロギング + PII 除外
- 3層通知システム（無通知 / トースト / モーダル）
- パフォーマンス目標値の達成（CI 自動測定）
- EV / OV コードサイニング
- Tauri Updater による自動アップデート + マイグレーション + ロールバック

### v1.0 では明示的に除外（v2.0 以降で検討）

- `.ani` 新規生成
- 簡易ペイント / ベクター編集
- ライブプレビュー（OS 全体への一時適用）
- 適用 Undo（直前テーマへの戻し、パニックボタンのみ）
- 自動切替トリガー（時刻 / アプリフォーカス / ホットキー切替）
- クラウド同期 / 自動定期バックアップ
- タグ / フォルダによる階層分類
- 他カーソル管理ツールの検知警告
- ja / en 以外の言語
- アプリ内 GitHub OAuth 自動 PR
- Microsoft Store 配布（v1.x で `.msi` → `.msix` を経て検討）
- Win10 22H2 未満のサポート
- RDP / Windows Server 環境

### リリース判定基準（Definition of Done）

1. すべての機能スコープが実装され、CI でグリーン
2. パフォーマンス目標値（§5）をすべて達成
3. **マニュアル受け入れテスト**: Win11 最新 (x64) と Win11 最新 (ARM64) の 2 環境で実施。
   - **Win10 22H2 はリリース前のマニュアルテスト対象外**。サポート OS としては公称するが、実機検証はコミュニティ報告ベースで v1.x 以降に対応する（既知のリスクとして許容）。
4. README + ユーザーガイド（ja / en）整備
5. EV / OV コード署名済みインストーラー（`.msi` x64 / `.msi` ARM64）公開
6. SmartScreen レピュテーション獲得（最低数十ダウンロードでの実測検証）
